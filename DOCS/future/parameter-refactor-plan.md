# План рефакторинга: работа с параметром сборки

## Статус

Отложено. Решение по реализации ещё не принято.

## Проблема

Сейчас плагин создаёт общий параметр `AssemblyParameter` через временный файл общих параметров. Хотелось бы упростить этот процесс и избавиться от костыля с временным файлом.

## Важное уточнение по Revit API

В контексте проектного документа (`Document`) через Revit API **можно создавать только общие параметры** (shared parameters). Параметры проекта (project parameters) без GUID программно создать нельзя. Это подтверждается, в том числе, ответом Jeremy Tammik:

> When you are in the context of a project document, you can only create shared parameters through the API, just like through the user interface.

В старых версиях Revit существовала перегрузка `BindingMap.Insert(string, ParameterType, Binding, BuiltInParameterGroup)`, но она всё равно создавала общий параметр под капотом. В Revit 2025+ эту перегрузку убрали, поэтому теперь нужно явно создавать `ExternalDefinition` через `DefinitionFile`.

## Варианты реализации

### Вариант 1. Оставить текущий подход с общим параметром

Создавать общий параметр `AM_Parameter` через временный файл общих параметров, но сделать код надёжнее:
- вынести имя параметра в константы,
- проверять существующий параметр по имени,
- проверять тип и привязку,
- автоматически дополнять привязку категорий,
- корректно восстанавливать `SharedParametersFilename`,
- группу параметров установить в «Общие» (`PG_GENERAL`).

Плюсы:
- Работает для всех версий Revit.
- Параметр появляется в диалоге Manage > Project Parameters.
- Фильтры видов работают корректно.

Минусы:
- Требуется временный файл общих параметров.
- Параметр имеет GUID и теоретически может конфликтовать при копировании между проектами.

### Вариант 2. Использовать уже имеющийся в проекте общий параметр

Вместо создания собственного параметра плагин может:
- искать в документе общий параметр с определённым именем (например, `AM_Parameter` или другое имя по конфигурации),
- если параметр найден — использовать его для записи имён сборок,
- если не найден — предложить пользователю создать его вручную через файл общих параметров или UI плагина.

Плюсы:
- Плагин не создаёт параметр самостоятельно.
- Нет проблем с временными файлами и GUID.
- Пользователь полностью контролирует параметр.

Минусы:
- Требуется предварительная настройка проекта.
- Пользователю нужно знать имя параметра и его тип.
- Сложнее для новичков.

### Вариант 3. Создать параметр через файл общих параметров проекта

Создать постоянный файл общих параметров `.txt` в составе плагина (например, в папке `%APPDATA%\AssemblingManager\shared-params.txt`). Использовать его для создания параметра `AM_Parameter`.

Плюсы:
- Нет необходимости создавать временный файл при каждом запуске.
- GUID параметра стабилен.

Минусы:
- Появляется дополнительный файл, который нужно сопровождать.
- Установщик/деплой должен копировать этот файл.
- Всё ещё общий параметр, а не параметр проекта.

## Рекомендации

Если цель — простота и автоматизация, лучше остановиться на **варианте 1** с улучшенной реализацией.

Если важно, чтобы плагин не создавал параметры без участия пользователя — рассмотреть **вариант 2**.

## Проверка вручную (контрольный список)

Когда реализация будет выбрана и выполнена, проверить в Revit:

- [ ] Плагин собирается без ошибок (`dotnet build build/AssemblingManager.sln -c Debug.R25`).
- [ ] `build-and-deploy.bat` выполняется успешно.
- [ ] В новом документе параметр создаётся автоматически.
- [ ] В диалоге Manage > Project Parameters виден параметр `AM_Parameter` (или выбранное имя).
- [ ] Тип параметра — `Text`, группа — `Общие`, привязка — к экземплярам.
- [ ] При повторном запуске команды параметр не дублируется.
- [ ] Значения параметра заполняются именами сборок.
- [ ] Фильтры видов корректно скрывают посторонние элементы.
- [ ] Если параметр существует с неправильным типом — плагин показывает понятную ошибку.
- [ ] Если в новой сборке появляются элементы новой категории — привязка дополняется.
