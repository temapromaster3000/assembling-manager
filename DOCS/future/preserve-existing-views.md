# Сохранение наработок проектировщика при повторном запуске

## Проблема

Сейчас плагин перед созданием видов удаляет старые виды с именами `{ИмяСборки}_План`, `{ИмяСборки}_Вид спереди` и т.д. Это приводит к потере оформления, которое проектировщик мог сделать на листах:
- размещение видов на листах слетает;
- настройки видимости, crop box, марки и размеры теряются;
- приходится оформлять заново.

## Цель

При повторном запуске плагина **не удалять** существующие виды и **не обновлять** их. Создавать виды только для новых сборок или для тех видов, которых ещё нет.

## Предлагаемый подход

### 1. Отказаться от удаления существующих видов

- Убрать вызов `_viewService.DeleteExistingViews(doc, assembly.Name)` из `OrchestratorService.GenerateViews`.
- Либо оставить метод `DeleteExistingViews` на будущее, но не вызывать его.

### 2. Проверять существование вида перед созданием

- Добавить в `ViewService` метод:
  ```csharp
  private bool ViewExists(Document doc, string viewName)
  ```
- Перед созданием каждого вида искать в документе вид с таким именем.
- Если вид существует — пропустить создание.
- Если не существует — создать.

### 3. Создавать только недостающие виды

- Для каждой сборки и каждого выбранного типа вида выполнять проверку.
- Если вид уже есть — ничего не делать.
- Если вида нет — создать по текущей логике.

### 4. Учитывать новые сборки

- Если в модели появилась новая `AssemblyInstance`, для неё создаются все выбранные виды.
- Если сборка удалена, её виды остаются в проекте — это приемлемо.

### 5. Отчёт о выполнении

- Добавить окно или лог с результатами:
  - список созданных видов;
  - список пропущенных (уже существующих) видов.
- Это поможет пользователю понять, что произошло.

## Открытые вопросы

### Переименование видов проектировщиком

Если проектировщик переименовал вид, например `{ИмяСборки}_Вид спереди` → `{ИмяСборки}_Фасад`, плагин не найдёт его по имени и создаст дубль.

Возможные решения:
- **Простое**: искать только по точному имени. Дубли возможны, но реализация простая.
- **Сложное**: добавить скрытый параметр или хранить в документе карту соответствия `AssemblyId → ViewId`. Это позволит находить вид независимо от имени.

### Изменение геометрии сборки

Если геометрия сборки изменилась, старые виды останутся со старым `SectionBox`. Это может быть как плюсом (сохраняется оформление), так и минусом (вид перестаёт охватывать сборку).

Решение: не обновлять существующие виды. Если пользователю нужно пересоздать — он удалит вид вручную и запустит плагин.

### Удалённые сборки

Если сборка удалена, виды для неё остаются в проекте. Можно добавить отдельную команду «Удалить виды удалённых сборок», но это отдельная задача.

## Файлы, которые затронет изменение

- `src/AssemblingManager.Revit/Services/ViewService.cs`
- `src/AssemblingManager.Revit/Services/OrchestratorService.cs`
- Возможно `src/AssemblingManager.Revit/Views/MainWindow.xaml` — для отображения отчёта.

## Контрольный список ручной проверки

1. Создать виды для нескольких сборок.
2. Разместить виды на листах, добавить марки/размеры.
3. Изменить одну из сборок (например, переместить элемент).
4. Повторно запустить плагин.
5. Убедиться, что уже оформленные виды не удалились и не сместились.
6. Убедиться, что для новой сборки создались все выбранные виды.
7. Убедиться, что для изменённой сборки старые виды остались без изменений.
8. Проверить, что плагин не падает с ошибкой при повторном запуске.

## Статус

Пока отложено. Реализовать после стабилизации основной логики создания видов.
