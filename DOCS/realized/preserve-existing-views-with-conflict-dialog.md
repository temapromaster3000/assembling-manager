# Сохранение наработок при повторном формировании видов (с диалогом конфликтов)

## Проблема

Сейчас плагин перед созданием видов удаляет все старые виды с именами `{ИмяСборки}_План`, `{ИмяСборки}_Вид спереди` и т.д. Из-за этого проектировщик теряет оформление на листах: размещение видов, марки, размеры, настройки видимости и crop box.

## Цель

При повторном запуске плагина не удалять существующие виды автоматически. Вместо этого:

- найти виды, которые уже существуют и могут быть перезаписаны;
- показать пользователю список таких конфликтов;
- дать возможность выбрать, какие виды заменить, а какие оставить без изменений;
- создавать только те виды, которых ещё нет;
- по завершении показать краткий отчёт.

## Поведение

### 1. Главное окно выбора видов

Без изменений. Пользователь выбирает, какие типы видов создавать (План, Разрезы, 3D), и нажимает «Сформировать виды».

### 2. Окно конфликтов

После нажатия «Сформировать виды» плагин анализирует документ: для каждой выбранной сборки и каждого выбранного типа вида проверяет, существует ли вид с ожидаемым именем.

Если такие виды найдены — показывается окно конфликтов со списком:

- каждая строка — один существующий вид (`{ИмяСборки}_Суффикс`);
- в строке галочка «Заменить» (по умолчанию снята);
- кнопка «Выбрать всё» — переключает все галочки;
- кнопки «OK» и «Отмена».

При нажатии «Отмена» плагин возвращает пользователя в главное окно выбора видов, ничего не создавая.

Если конфликтов нет — окно конфликтов не показывается, сразу выполняется создание видов.

### 3. Создание видов

После нажатия «OK»:

- виды, отмеченные для замены, удаляются и создаются заново;
- виды, не отмеченные для замены, остаются без изменений;
- виды, которых не было, создаются;
- фильтр сборки пересоздаётся, если для сборки заменяется хотя бы один вид, и применяется ко всем видам этой сборки (включая пропущенные).

### 4. Отчёт

После завершения работы показывается окно-отчёт:

- количество созданных видов;
- количество заменённых видов;
- количество пропущенных видов;
- время работы плагина.

Нулевые значения не выводятся. Время считается с нажатия «Сформировать виды» (или «OK» в окне конфликтов, если оно было) до завершения операции.

## Технические изменения

### Модели

- Расширить `ViewCreationOptions` (или создать отдельный объект), чтобы передавать из UI в сервисы выбор пользователя: какие именно виды заменять.
- Добавить модель результата работы: `ViewCreationResult` с полями `Created`, `Replaced`, `Skipped`, `Elapsed`.

### Сервисы

- `ViewService`:
  - добавить метод поиска существующих видов по именам;
  - удалить вызов `DeleteExistingViews` из публичного API или заменить на удаление только выбранных видов.
- `OrchestratorService`:
  - убрать безусловный вызов `_viewService.DeleteExistingViews`;
  - перед созданием каждого вида проверять, существует ли он;
  - если вид существует и выбран для замены — удалить старый и создать новый;
  - если вид существует и не выбран для замены — пропустить;
  - если вида нет — создать;
  - пересоздавать фильтр и применять его ко всем видам сборки, если для неё заменяется хотя бы один вид.
- `FilterService`:
  - добавить метод удаления существующего фильтра по имени перед созданием нового;
  - применять фильтр к списку видов.

### UI

- Добавить `ConflictDialog.xaml` — окно конфликтов со списком и галочками.
- Добавить `ReportDialog.xaml` — окно отчёта.
- В `CreateAssemblyViewsCommand`:
  - открывать окно конфликтов перед началом транзакции;
  - замерять время;
  - открывать окно отчёта после успешного выполнения.

## Открытые вопросы и ограничения

### Переименование видов пользователем

Если проектировщик переименовал вид, плагин его не найдёт и создаст дубль. Это сознательное ограничение: отслеживание видов после переименования вручную не реализуется.

### Изменение геометрии сборки

Если геометрия сборки изменилась, а вид пропущен — он останется со старым crop box / section box. Чтобы обновить, пользователь должен выбрать замену.

### Удалённые сборки

Виды удалённых сборок остаются в проекте. Удаление таких виды — отдельная задача.

### Параметр AssemblyParameter

При пропуске всех видов для сборки параметр **не** перезаписывается. Если состав сборки изменился, пользователь должен заменить хотя бы один вид, чтобы параметр обновился.

## Контрольный список ручной проверки

1. Создать виды для нескольких сборок.
2. Разместить виды на листах, добавить марки/размеры.
3. Изменить одну из сборок (например, переместить элемент).
4. Повторно запустить плагин.
5. Убедиться, что окно конфликтов показывает существующие виды.
6. Выбрать замену для части видов, остальные пропустить.
7. Убедиться, что заменённые виды создались заново.
8. Убедиться, что пропущенные виды сохранили оформление на листах.
9. Убедиться, что для новой сборки создались все выбранные виды.
10. Проверить, что фильтр корректно применяется к заменённым видам.
11. Проверить, что отчёт показывает правильные цифры и время.
12. Проверить, что при нажатии «Отмена» в окне конфликтов возвращаемся в главное окно.
